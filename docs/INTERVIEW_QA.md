# 面试 Q&A：AI 秋招投递决策助手

## 1. 为什么做这个项目？

回答思路：秋招投递中 JD 阅读、岗位判断、简历修改和进度管理非常碎片化。我希望用一个 AI 工具把这些动作串起来，提高效率，也沉淀可复盘的数据。

## 2. 目标用户是谁？

回答思路：主要是 2027 届准备秋招或实习的学生，尤其是投 AI 产品、产品经理、项目管理、数据分析、运营等岗位的人。

## 3. 你怎么定义 MVP？

回答思路：第一阶段先保证本地可跑通：上传简历、粘贴 JD、规则匹配、保存投递记录、管理状态和看板复盘。LLM 作为第二阶段增强。

## 4. AI 能力具体落在哪里？

回答思路：AI 不是泛泛聊天，而是把简历和 JD 转成结构化分析，包括匹配度、证据、缺口、建议、面试亮点和风险。

## 5. Prompt 怎么设计？

回答思路：Prompt 明确限定输入边界，只能基于原始简历和 JD，要求输出 JSON，并强调不能虚构项目、公司、学校、奖项、指标。

## 6. 怎么防止简历造假？

回答思路：一是 Prompt 约束不能虚构；二是 matched_evidence 必须能对应简历证据；三是页面和导出简历都加免责声明，要求用户人工核对。

## 7. 怎么评估岗位匹配度？

回答思路：LLM 版综合 JD 要求和简历证据；fallback 版基于 JD 关键词与简历文本重合度，加上简历/JD 完整度。匹配度用于排序和决策参考，不替代人工判断。

## 8. 为什么要保留规则 fallback？

回答思路：作品集 Demo 需要稳定演示。即使没有 API Key、模型限流或网络异常，用户也能完成核心流程。

## 9. LLM 输出为什么用 JSON？

回答思路：JSON 便于页面结构化展示、数据库保存、后续看板分析和异常处理，比纯文本更适合产品化。

## 10. 如果模型输出不是合法 JSON 怎么办？

回答思路：先做容错解析，包括代码块和文本中的 JSON；如果仍失败，展示原文和友好提示，并自动切换规则 fallback。

## 11. 怎么优化成本？

回答思路：只在用户点击分析时调用模型；输入截断到合理长度；简历生成 v0.2 先用规则和分析结果生成，不额外调用第二次 LLM。

## 12. 怎么优化延迟？

回答思路：本地解析和保存很快；LLM 调用只走一次；无 Key 或失败时快速 fallback。后续可加缓存和异步任务。

## 13. 为什么选择 Streamlit？

回答思路：目标是快速验证产品流程和作品集展示，Streamlit 能低成本搭建可交互 Demo，适合 vibe coding 的 MVP 阶段。

## 14. 数据库为什么用 SQLite？

回答思路：本地工具不需要复杂后端，SQLite 部署成本低、足够支持投递记录增删改查和看板分析。

## 15. 数据看板有什么价值？

回答思路：它把投递从一次性动作变成可复盘过程，帮助用户看到投递方向、匹配度结构、JD 高频能力和待跟进行动。

## 16. 定制简历生成怎么保证真实？

回答思路：只重组和强化原简历已有经历，不新增公司、岗位、时间、奖项或指标；无法确认的信息保留原文。

## 17. 你在项目中做了哪些产品取舍？

回答思路：先保证端到端闭环和稳定 fallback，再追求 LLM 质量；先做 Top 20 能力条形图，不引入词云依赖，降低环境复杂度。

## 18. 这个项目和普通 ChatGPT 问答有什么区别？

回答思路：它把 AI 能力嵌入完整业务流程，有结构化输入输出、状态管理、数据库记录、导出和看板复盘，不只是一次性对话。

## 19. 后续会怎么迭代？

回答思路：会增加 LLM 简历生成、多版本简历管理、岗位优先级、面试复盘、模型调用成本与成功率监控。

## 20. 如果给真实用户使用，还需要补什么？

回答思路：需要更严格的隐私说明、文件删除能力、用户数据隔离、模型供应商选择、错误监控，以及更完整的测试覆盖。
